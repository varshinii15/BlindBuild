BlindBuild

Participants are given an already deployed backend. API documentation will not be provided. Teams must identify useful endpoints through exploration and integrate them into their frontend. It is not necessary to discover every endpoint.

Round 2: Problem Brief for API Architecture

Participants must design the backend API structure required to support frontend workflows. Each question below defines application behavior, user actions, and design expectations.

1. Event Registration & Ticket System

Problem Statement

This module manages participant onboarding and event sign-ups, while issuing unique digital tickets to support smooth movement from registration to venue entry.

Scenario Context

The experience should cover account-level setup, event discovery, sign-up validation, registration tracking, ticket identity generation, and cancellation handling when required.

Task Focus

Design the complete API workflow, including resources, methods, payloads, and response formats.

Concepts Tested

API architecture, workflow design, and system thinking.

2. Workshop Slot Booking System

Problem Statement

This module supports workshop scheduling and seat reservation with time-slot awareness, ensuring fair allocation and preventing overbooking.

Scenario Context

Participants should be able to browse workshop sessions, understand slot availability, reserve preferred timings, update decisions when needed, and verify their booking state.

Task Focus

Design APIs that manage nested workshop-slot relationships and booking lifecycle actions.

Concepts Tested

REST resource structure, nested resources, and end-to-end flow design.

3. Convenor & Event Management Hub

Problem Statement

This organizer-facing module supports operational control during events, including participant validation, attendance handling, and event-level result management.

Scenario Context

Convenors should be able to verify ticket authenticity, mark and revise attendance, retrieve participant details, generate badge outputs, and manage winner records in a traceable flow.

Task Focus

Design organizer-oriented APIs that support real-time event control and outcome management.

Concepts Tested

Operational workflow design, validation logic, and event data governance.

4. Event Feedback & Rating System

Problem Statement

This module captures participant experience after events through written feedback and rating input, helping organizers evaluate quality and satisfaction.

Scenario Context

Responses should remain manageable over time so entries can be reviewed, refined, removed when needed, and aggregated into a meaningful performance overview.

Task Focus

Design CRUD-ready feedback APIs and rating aggregation behavior with predictable response formats.

Concepts Tested

CRUD API design, data aggregation, and post-event analytics flow.

5. Lost & Found Campus Portal

Problem Statement

This module provides a structured recovery process for misplaced items, from initial reporting to validated return across campus event spaces.

Scenario Context

The flow should support item reporting, report matching, claim tracking, admin-level decisions, and final closure with accountability at each stage.

Task Focus

Design a multi-actor API workflow with clear claim states, role responsibilities, and status transitions.

Concepts Tested

Workflow modeling, resource linking, and status management.

Instruction: Derive the complete backend API contract for each question without relying on pre-listed endpoint patterns. Your design should include resources, HTTP methods, required payload fields, and response structures.
